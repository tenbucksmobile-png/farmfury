This is the round-by-round debugging/audit log for the Unity port (`unity/`), moved out of
`CLAUDE.md` to keep that file a lean architecture reference. Read this when you need the
reasoning/root-cause behind a current value or convention in `CLAUDE.md` ‚Äî it explains *why*,

## Audit Findings (2026-07-01)

A full audit was run against CLAUDE.md, the GDD, and the actual project state. This block is the
standing summary of that audit ‚Äî re-verify anything here before relying on it if significant
time has passed.

**Critical bug ‚Äî L02‚ÄìL06 spawn floating in mid-air.** `LevelDataGenerator.cs` still carries a header comment for the *pre-rebuild* coordinate system (`ground = -2.5`, `launcher X = -5.5`) and L02‚ÄìL06 were generated under that system (block/robot Y ‚âà ‚àí1.5 to ‚àí2.3). The live scene was recalibrated to `ground Y = ‚àí6.60`, `launcher X = ‚àí2.327` (see Phase 4), but **L02‚ÄìL06's baked Y values were never migrated** ‚Äî `LevelLoader.SpawnBlock`/`SpawnRobot` place objects at the raw stored position with no offset applied, so these five levels currently spawn their entire structure ~4.4‚Äì4.9 units above the actual ground, disconnected from it, and their X positions are calibrated to the old launcher position rather than the current one. **Only L01 (hand-edited directly in the asset, not via the generator) uses correct current-system coordinates.** L02‚ÄìL06 need their baked positions recalculated (roughly: old Y ‚àí 4.1, and X re-checked against the new launcher at X=‚àí2.327) before they're playable. Still true as of the latest check (2026-07-19, Round 12) ‚Äî `LevelDataGenerator.cs`'s L02‚ÄìL06 `Make()` calls still use Y ‚âà ‚àí0.9 to ‚àí2.3, the old system. Treat L02‚ÄìL06 as non-functional despite existing as assets.

**`LevelDataGenerator.cs` L01 landmine ‚Äî RESOLVED 2026-07-01.** The generator's L01 entry was rewritten to match reality and extended with a 6-bale hay-pile obstacle (see "L01 gameplay pass" below); it's now safe to run **FarmFury ‚Üí Generate All Level Data** again ‚Äî it reproduces the current L01 design instead of destroying it. L02‚ÄìL06 are untouched and still on the old coordinate system (see bug above) ‚Äî regenerating will not make them worse, but won't fix them either.

**L01 gameplay pass ‚Äî 2026-07-01.** Fixed two real bugs found while wiring up the drag/aim/pass-through mechanics against the user's request:
1. `CatapultLauncher.DrawTrajectory()` was fully implemented but **never called** ‚Äî the dotted aiming-arc preview (`_trajDotRenderers`, built in `Awake()`) never actually drew during drag. Fixed: now called from the drag-hold branch of `HandleInput()`, and its start position corrected from the fixed rest `_launchPoint` to `BucketWorldPos(_dragAngle)` (the actual point `Fire()` spawns the animal from).
2. `LevelLoader.SpawnBlock()` never applied `BlockSpawnData.passThrough` / `.healthOverride` / `.massOverride` to the spawned block ‚Äî `WoodBlock.cs`'s own comment claimed LevelLoader did this; it didn't. This silently broke the Cluck pass-through mechanic for any level using it. Fixed: added `BlockBase.ApplyOverrides(health, mass)`, called from `SpawnBlock()` along with setting `WoodBlock._passThrough` directly.

Added a new `BlockType.Haybale` (enum value 2, `LevelData.cs`) and a `HaybaleBlock.prefab` ‚Äî mechanically identical to `WoodBlock` (same component, `SceneSetup.EnsureHaybaleBlockPrefab()` creates it if missing), just wired to render `Assets/Sprites/Environment/World1Props/Haybail.png` instead of the generic wood-plank art. **No new art was needed** ‚Äî `Haybail.png` was already imported (used for the decorative scenery), and `BlockBase`'s destruction visuals (procedural crack overlays at 50%/25% health, tint shift, 4 fading fragments) are entirely sprite-agnostic, so haybale destruction "just works" the same as any other block once the sprite is wired.

**L01 now has an actual obstacle**, not just a bare robot: 4 haybale blocks (`passThrough=true`, `healthOverride=60`, `massOverride=3` ‚Äî lighter/weaker than standard wood so the tutorial stays easy). Positions were originally guessed, then corrected in the round-2 pass below to match the user's real hand-placed decorative haybales exactly. Requires the user to run **FarmFury ‚Üí Wire Scene References** once (materializes `HaybaleBlock.prefab` and wires `_haybalePrefab` into `LevelLoader`) before it'll spawn correctly.

**Visual-feedback round 2 ‚Äî 2026-07-01, from user screenshot.** Play-test report: HUD cards overlapping the screen edge, trebuchet arm launch looking wrong, no visible Cluck in flight, HarvesterRobot floating in mid-air instead of nestled behind the hay, plus a request to use the real hay-pile positions instead of the guessed ones. Diagnosed and fixed by reading `Game.unity` directly (no render/play-test access from this side):
- **Robot floating ‚Äî root cause found.** `RobotEnemy.Awake()` sets `BoxCollider2D.size=(1,1)`, sized for the default `(0.6,0.9)` scale. L01's HarvesterRobot uses a custom `(4.3565,4.69371)` visual scale for a bigger, more imposing enemy, but nothing shrunk the collider to compensate ‚Äî it inherited the full scale, becoming a ~4.36√ó4.69-unit hitbox that overlaps the ground (Y=‚àí6.60) by roughly 2 units at spawn. Unity's physics forcefully separates deep overlaps on the first frame, launching the robot upward. Fixed in `LevelLoader.SpawnRobot()`: when a custom scale is applied, the collider size is now re-derived (`0.6/scaleX, 0.9/scaleY`) to keep the world-space hitbox pinned at the original 0.6√ó0.9 regardless of visual scale.
- **Trebuchet likely invisible ‚Äî root cause found.** The hand-placed `Trabuchet_Body`/`Trabuchet_Arm`/`Trabuchet_Swing` scene GameObjects were never given a `sortingOrder` (default 0), tied with Ground/scenery (also ~0). `Trabuchet_Body` additionally sits at `Z=2`, deeper than Ground's `Z=0`. Unity's default 2D sort falls back to camera distance on sortingOrder ties, so the trebuchet was very likely rendering behind the terrain. Fixed in `SceneSetup.WireLauncher()`: now sets sortingOrder 3/4/5 (body/arm/swing) and flattens Z to 0. Self-heals on the next **Wire Scene References** run.
- **Hay pile repositioned to the user's real placement.** Read the exact transforms of the 4 hand-placed decorative `Haybail`/`Haybail (1-3)` GameObjects directly from `Game.unity` (positions ~X 3.65‚Äì4.31, Y ‚àí4.88 to ‚àí5.65) and used those exact coordinates for the gameplay `HaybaleBlock` entries, replacing the earlier guessed grid. **The 4 decorative scene GOs should be deleted** once this is live (done in Round 6, below).
- **HUD safe-area gap.** Score/bird-queue/pause elements were anchored to the raw screen rect with no accounting for device notches/rounded corners. Added a `SafeArea` RectTransform wrapper (`Screen.safeArea`-driven) in `HUDController.BuildCanvas()`.
- **Trebuchet arm position/scale not sticking in the Scene view ‚Äî by design, not a bug.** `CatapultLauncher.DrawArmAt()` forcibly repositions arm/swing sprites to the pivot-derived position every frame at runtime. Manually dragging in the Scene view outside Play mode is edit-time-only reference positioning. (Note: this whole trebuchet-arm subsystem was removed in the 2026-07-02 cannon swap, below ‚Äî kept here for historical context only.)

**Broken scene PPtr + sortingOrder tie ‚Äî 2026-07-04.**
- **`Game.unity` had a broken PPtr** after the Cannon-duplicate deletion above ‚Äî removing the GameObject/SpriteRenderer/Transform blocks by hand missed that Unity's internal `SceneRoots.m_Roots` array still listed that Transform's fileID. **Lesson: when hand-deleting a root-level GameObject from a `.unity` file, also grep for its component fileIDs across the whole file afterward.**
- **All 8 animal subclasses overrode `_sr.sortingOrder = 4` in their own `Awake()`**, tying exactly with `FarmCannon`'s `sortingOrder=4` ‚Äî ties fall back to Z-distance, so birds could render behind the cannon both loaded and in flight. Fixed to `sortingOrder = 5` at the time (moved to 6 in Round 7).
- Added `AnimalBase.SetInFlightPose()` ‚Äî sets `_sprInFlight` without touching physics/launch state, used by `PrepareNextBird()` instead of the seated loaded pose.

**Round 4 ‚Äî 2026-07-04. Smoke texture + robot Y-clipping (both confirmed by computation); chicken/fire still unresolved.**
- **Smoke rendered as squares.** `BuildSmokeParticles()` assigned a shader Material with no texture ‚Äî a bare quad with no texture renders as a plain square. Added `MakeSoftCircleTexture()` (soft radial falloff) as `mat.mainTexture`.
- **Robot "still outside the screen" ‚Äî X was never the problem; camera math was.** Confirmed via a scenery object visible at X=7.24 that the camera's horizontal range covers X=4.8. Computed the vertical viewport instead: camera rest Y=-2, orthoSize=4.5 ‚Üí visible Y range -6.5 to +2.5. Ground line is at Y=-6.60 ‚Äî only 0.1 units below the camera's own bottom edge. Measured the robot sprite's real art bounds and found it extended 0.23 units below the camera's bottom edge; raised to Y=-5.8215. **This camera/ground-line relationship (0.1u clearance) affects any future tall ground-standing enemy in World 1, not just this one robot.**

**Round 5 ‚Äî 2026-07-05. `_eggPrefab` UnassignedReferenceException.** Confirms firing/rendering from Round 4 actually works (the crash trace shows a full launch ‚Üí flight ‚Üí tap-to-trigger sequence; the failure was purely in egg spawning). **Round 9 ‚Äî 2026-07-12. SECOND, deeper root cause: character sprite folders were misnamed, so `SpriteWiring.cs` was wiring nothing.** After the Round 8 fix, the bird rendered as "a yellow dot" (AnimalBase's procedural-circle fallback, tinted by CluckAnimal when `!HasRealSprites`). **Round 12 ‚Äî 2026-07-18/19. L01 robot free-fall + floating haybails (both root-caused); mobile aim-drag hypersensitivity fixed; MatchUp and WorldMap "baked mockup + duplicate code overlay" bugs found and fixed; new pin-free SunriseMeadows.png generated and all 18 path positions re-traced from real pixel data; robot/haybail hit-reaction art added.**
- **HarvesterRobot "starts right, then falls" ‚Äî real bug.** `RobotEnemy` never set `Rigidbody2D.bodyType` ‚Äî the prefab shipped `Dynamic` (Unity default), unlike every `BlockBase` type, which explicitly sets `Static`. Every robot free-fell from its spawn Y under gravity from frame one. Fixed by setting `_rb.bodyType = RigidbodyType2D.Static` in `RobotEnemy.Awake()` ‚Äî the damage formula already special-cases a `Static` partner, so only the fall stopped, not hit detection. **Current live behavior.**
- **L01 haybail pile "still too high" ‚Äî distinct from the Round 10 stack-gap fix.** Round 10 correctly fixed the *internal* gap between the top bale and base row. What nobody had checked: the base row's Y values were copied from the user's decorative props without validating against the ground line ‚Äî the whole pile floated ~0.63 units above where it should sit. Fixed by shifting all 4 Y values down by 0.627u ‚Äî preserves the base row's intentional unevenness while anchoring the lowest bale to the ground. **Current live positions are in `LevelDataGenerator.cs`/CLAUDE.md.**
- **Mobile aim-drag "feel" ‚Äî root cause measured, not guessed.** The loaded bird sits only ~0.055 world units from the aim pivot, so the first sliver of pointer movement swings the drag angle across most of the 50¬∞ pull range ‚Äî worse on touchscreens. Fixed with `MinAimRadius = 0.6f`: the drag angle freezes until the pointer has moved at least that far. Confirmed `Pointer.current` was already correctly covering touch ‚Äî this was never an input-API gap.
- **MatchUpScreen double-frame bug** ‚Äî `_matchUpBackgroundSprite` was wired to `MatchUp_Background.png`, which turned out to bake in the sky, both picture frames, the chicken/robot, and the VS graphic all in one image ‚Äî not a plain backdrop as its own comment claimed. The real modular cards drawn on top produced two overlapping frames. Fixed by rewiring to `Background_SkyV1.png` (plain sky, nothing else baked in).
- **WorldMapController had the exact same "baked mockup used as live background" bug** for pins and NEXT LEVEL/Home buttons (title already fixed in Round 11). `SunriseMeadows.png` baked in 18 placeholder pins plus both buttons; the code drew separate versions on top of all three.

**Round 16 ‚Äî 2026-07-27. L02‚ÄìL04 migrated from the pre-rebuild coordinate system to the current one; L05‚ÄìL06 deliberately left untouched.**
- **The bug:** `LevelDataGenerator.cs`'s `L02_StoneWall`/`L03_TheTower`/`L04_EggPractice` `Make()` calls were still authored against the OLD system (ground Y=-2.5, launcher X=-5.5) from before the 2026-07-01/02 rebuild that moved the ground to Y=-6.60 and the launcher to X=-2.327. Since `LevelLoader.SpawnBlock`/`SpawnRobot` apply zero offset at spawn time, every block/robot in these three levels floated ~4.4 units above the actual ground ‚Äî this had been the single highest-leverage blocker on World 1 since it was first flagged.
- **The fix:** a single uniform rigid-translation delta, `dx = -2.327 - (-5.5) = +3.173`, `dy = -6.60 - (-2.5) = -4.10`, applied identically to every block AND robot position in all three levels (`new = old + delta`). A pure delta was chosen over hand-retuning each object specifically because it's a translation ‚Äî it mathematically preserves every relative relationship the original design already had: bottom rows that rested exactly on the old ground line still rest exactly on the new ground line, inter-block gaps are unchanged, and `L03`'s second robot (perched on top of its tower, not the ground) is still perched on top of the tower post-migration.
- **Why this matters for Wood/Stone blocks specifically (not just cosmetics):** these blocks spawn `Static` and only the *first* `TakeDamage()` anywhere in the level wakes ALL of them to `Dynamic` via `WakeAllStaticBlocks()` (`BlockBase.cs`). If migration had left even a small gap between a bottom row and the true ground, that gap would be invisible at level start (Static bodies don't respond to gravity) and then the entire structure would visibly free-fall into it the instant any block anywhere took its first hit ‚Äî a jarring, level-wide "drop" triggered by an unrelated shot. A uniform delta keeps every bottom row exactly ground-touching (since it was exactly ground-touching before the shift too), so this can't happen. Robots were never at risk either way ‚Äî `RobotEnemy.Awake()` force-sets `bodyType = Static` permanently and never wakes, regardless of any gap.
- **Accepted, pre-existing trade-off, not something introduced by this fix:** ground-resting objects still clip their bottom ~0.1 units off the bottom of the camera's rest-view (`Y -6.5` to `+2.5` vs. true ground at `Y=-6.60`) ‚Äî the same documented constraint CLAUDE.md's Coordinate System section already calls out generally ("any sizeable ground-standing sprite will clip the bottom of the screen"). No extra lift was applied to compensate, since lifting blocks off the true ground line would reintroduce the free-fall-gap risk described above; this is a materially smaller clip than the ~0.23-unit one L01's hay pile had before its own fix (Known Issues, 2026-07-26), since most of a stacked block/robot silhouette (0.4-0.9 units tall) sits comfortably above the visible line regardless.
